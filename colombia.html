🏦 PSE
Core
Full name Document number Email
Common
Phone number Document type Bank
Additional
Address City Account type
Merchant side pattern

Most PSE merchant forms request Full name + one identity field (Document number or Email). The 4bill-mediated merchants (bc.game, Brazino 777) show empty merchant sides — they delegate entirely to the provider. Direct integration merchants (1xbet, 20bet) ask for Bank and document fields themselves.

Provider side pattern

Provider APIs consistently require Bank + identity fields (Full name, Document type, Document number, Email, Phone number). This set appears in supefina.net, smartfastpay.com, and starpago flows. Simpler providers (4bill) only ask for Full name + Phone number + Email.

Recommended Minimum Field Set
Bank Full name Email Document type Document number
To build a low-friction PSE flow, collect Bank, Full name, Email, Document type, and Document number. Bank selection is essential — PSE routes the payment based on the user's bank. Phone number and Address should be added only if the specific PSP requires them. City appears provider-specific and is not a core field.

Merchant side pattern

Direct push flows (without PSP) uniformly collect identity + phone data. The 4bill-mediated routes show minimal or no merchant-side fields, with 4bill's provider API always requesting Full name + Phone number + Email. The Nequi p2p route is a manual flow requiring screenshots — not scalable.

Provider side pattern

4bill and 4bill Payzen (QR) consistently ask for Full name + Phone number + Email via their API. TRX point asks for Document type (and sometimes Document number) on the provider side. Since Nequi is phone-linked, Phone number appears critical at the API layer.

Recommended Minimum Field Set
Full name Phone number Document type Document number
For Nequi, Phone number is the core identifier — the wallet is phone-linked. Full name, Document type, and Document number appear consistently across flows. Email is useful but not universal. Avoid screenshot-based flows in production.

Merchant side pattern

4bill-mediated merchants (1win, Jugabet, Brazino777) show minimal or empty merchant sides. PSP-integrated providers (supefina, smartfastpay) request a richer set. P2P routes require manual proof (bank account number, screenshot) — not suitable for automated flows.

Provider side pattern

4bill API asks for Full name + Phone number + Email. supefina.net and smartfastpay.com APIs ask for Full name + Document type + Document number + Email + Phone number — a richer set. P2P has no observed provider-side API.

Recommended Minimum Field Set
Full name Phone number Email
For Bancolombia via 4bill (the most common observed provider), Full name + Phone number + Email appears sufficient. If using supefina.net or smartfastpay.com, add Document type and Document number. Avoid P2P flows for automated processing.

4bill for Daviplata shows an empty merchant side — consistent with its pattern of delegating to its own API. Daviplata is phone-linked — Phone number is the core identifier. Full name appears in all non-empty flows. Document fields and Address are provider-specific (Direct PSE / pagsmile). Start with Full name + Phone number and extend based on the provider chosen.
