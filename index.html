Payment Method Findings
Field analysis per payment method — based only on observed data. Helps identify the minimum field set needed to build a similar flow with minimal friction.
🏦 PSE
Fields consistently requested (merchant side)
  • Full name — appears across most merchant forms
  • Document number — requested by majority of merchants
  • Email — present in most PSE merchant forms
  • Phone number — common, especially where no PSP
  • Bank — required by Direct pay and 4bill merchant flows
Fields sometimes requested (merchant side)
  • Document type — mainly smartfastpay and starpago flows
  • City — only pay.trx-point.com (megapari, quotex)
  • Address — direct pay and smartfastpay flows only
  • Account type — only co-payer.erjqport.com provider side
Merchant side pattern

Most PSE merchant forms request Full name + one identity field (Document number or Email). The 4bill-mediated merchants (bc.game, Brazino 777) show empty merchant sides — the merchant appears not to collect extra fields, delegating entirely to the provider. Direct integration merchants (1xbet, 20bet) ask for Bank and document info themselves.

Provider side pattern

Provider APIs consistently require Bank + identity fields (Full name, Document type, Document number, Email, Phone number). This set appears in supefina.net, smartfastpay.com, and starpago flows. Simpler providers (4bill) only ask for Full name + Phone number + Email on their API side.

Recommended Minimum Field Set
Bank Full name Email Document type Document number
To build a low-friction PSE flow, collect Bank, Full name, Email, Document type, and Document number. Bank selection is essential — PSE routes the payment based on the user's bank. Phone number and Address should be added only if the specific PSP requires them. City appears to be provider-specific and is not a core field.
📱 Nequi
Fields consistently requested (merchant side)
  • Full name — present in the majority of direct push merchant forms
  • Phone number — core to Nequi (phone-based wallet)
  • Document type — common in direct push flows
  • Document number — appears alongside document type
Fields sometimes requested (merchant side)
  • Email — present in several but not all direct push forms
  • Screenshot / Comment — only in Nequi p2p (manual flow)
Merchant side pattern

Direct push flows (without PSP) uniformly collect identity + phone data. The 4bill-mediated routes show minimal or no merchant-side fields, with 4bill's provider API always requesting Full name + Phone number + Email. The Nequi p2p route is a manual flow requiring screenshots — not scalable.

Provider side pattern

4bill and 4bill Payzen (QR) consistently ask for Full name + Phone number + Email via their API. TRX point asks for Document type (and sometimes Document number) on the provider side. Since Nequi is phone-linked, phone number appears critical at the API layer.

Recommended Minimum Field Set
Full name Phone number Document type Document number
For Nequi, Phone number is the core identifier — the wallet is phone-linked. Full name, Document type, and Document number appear consistently across flows. Email is useful but not universal. Avoid screenshot-based flows in production as they require manual processing.
💳 Bre-b
Fields consistently requested (merchant side)
  • Full name — present in the one merchant form with data (Jugabet via 4bill→cobre)
  • Phone number — present alongside Full name
Fields sometimes requested (merchant side)
  • No additional fields observed — most rows show empty merchant sides
Merchant side pattern

Bre-b data is sparse. Most merchant-side cells are empty, suggesting either the merchant collects data before hand-off or the flow is redirect-based. The only filled merchant side (Jugabet via 4bill→cobre) shows Full name + Phone number.

Provider side pattern

The 4bill→cobre provider consistently requests Full name + Phone number + Email at the API level. topup.com.co shows no observed provider-side fields — data may be missing or the flow is simpler.

Recommended Minimum Field Set
Full name Phone number Email
Based on limited data, Full name + Phone number appear to be the core fields for Bre-b merchant flows. Email is likely needed at the provider API level (consistently requested by 4bill→cobre). Data coverage for this method is thin — treat these recommendations with caution.
🏛️ Bancolombia
Fields consistently requested (merchant side)
  • Full name — observed in Jugabet (4bill) and supefina/smartfastpay flows
  • Phone number — present in supefina and smartfastpay merchant forms
  • Email — present in supefina and smartfastpay merchant forms
Fields sometimes requested (merchant side)
  • Document type / number — only smartfastpay and supefina merchant forms
  • Bank number, Screenshot, TX id, Comment — P2P (manual) flows only
Merchant side pattern

4bill-mediated merchants (1win, Jugabet, Brazino777) show minimal or empty merchant sides. PSP-integrated providers (supefina, smartfastpay) request a richer set. P2P routes require manual proof (bank account number, screenshot), not suitable for automated flows.

Provider side pattern

4bill API asks for Full name + Phone number + Email. supefina.net and smartfastpay.com APIs ask for Full name + Document type + Document number + Email + Phone number — a richer set. P2P has no observed provider-side API.

Recommended Minimum Field Set
Full name Phone number Email
For Bancolombia via 4bill (the most common observed provider), Full name + Phone number + Email appears sufficient. If using supefina.net or smartfastpay.com, add Document type and Document number. Avoid P2P flows for automated processing.
📲 Daviplata
Fields consistently requested (merchant side)
  • Full name — observed in direct integration (melbet via Direct PSE)
  • Phone number — core field for this phone-linked wallet
Fields sometimes requested (merchant side)
  • Email — only in Direct PSE flow (melbet)
  • Document type / number — only in Direct PSE (melbet) and P2P (melbet)
  • Address — in Direct PSE flow
  • Screenshot / Comment — P2P manual flow
Merchant side pattern

4bill for Daviplata shows an empty merchant side — consistent with 4bill's pattern of delegating to its own API. Direct PSE integration (melbet) requests a full set of identity fields. api.lamapay.org/ p2p requests only Full name on the merchant side; the standalone P2P route for 1xbet requests Full name + Phone + Screenshot — a manual verification flow.

Provider side pattern

4bill API requests Full name + Phone number + Email. checkout.pagsmile.com requests a full identity set including Address. P2P and Direct PSE entries show no provider-side API data, suggesting direct bank-side integration.

Recommended Minimum Field Set
Full name Phone number
Daviplata is phone-linked — Phone number is the core identifier. Full name appears needed across all non-empty flows. Document fields and Address are provider-specific (Direct PSE / pagsmile). Start with Full name + Phone number and extend based on the provider chosen.
📷 Tpaga (QR)
Fields consistently requested (merchant side)
  • Document type — present in both P2P merchant rows (1win and melbet)
  • Document number — present alongside document type
Fields sometimes requested (merchant side)
  • Full name — only in melbet's P2P row (more complete form)
  • Email, Phone number, Address — only in melbet's P2P row
Merchant side pattern

Very limited data. Only P2P-routed merchants (1win and melbet) have observed merchant-side fields. 1win shows a minimal set (Document type + Document number), while melbet's form appears more complete. checkout.pagsmile.com / Binomo shows no fields on either side.

Provider side pattern

No provider-side fields observed in this payment method. checkout.pagsmile.com shows both sides empty for Tpaga (QR), which may indicate incomplete data capture at the time of research.

Recommended Minimum Field Set
Document type Document number
Based on limited data, Document type and Document number appear consistently for QR-based flows. However, Tpaga (QR) data coverage is very thin — treat this recommendation with caution and verify directly with the PSP before building.
🔄 Movii (PSE)
Fields consistently requested
  • Full name — observed in Direct PSE (melbet) form
  • Email — present in Direct PSE flow
  • Phone number — present in Direct PSE flow
  • Document type / number — in Direct PSE (melbet) form
Notes
  • Only 2 merchant rows observed — very limited coverage
  • 4bill (1win) shows empty merchant side, provider side Full name + Phone + Email
  • Appears to combine PSE routing with wallet-layer processing
Merchant side pattern

The 4bill route shows no merchant-side fields. The Direct PSE route (melbet) shows a full identity set including Address. Movii (PSE) appears to behave similarly to standard PSE in terms of required data.

Provider side pattern

Only 4bill shows provider-side data: Full name + Phone number + Email. Consistent with 4bill's standard API requirement across all payment methods.

Recommended Minimum Field Set
Full name Email Phone number
Extremely limited data for Movii (PSE) — only 2 providers and 2 merchants observed. Treat PSE field requirements as a baseline, then adjust based on provider. The 4bill API set (Full name + Phone + Email) appears as the practical minimum for this method.
Data Quality Notes
Notes on normalization and data limitations.
Provider names were normalized where clearly the same: h5.supefina.net and supefina.net → supefina.net; checkout.smartfastpay.com, Smartfastpay → smartfastpay.com; p2p (standalone, no domain) → P2P. Provider names containing a domain are never replaced with a generic label — api.lamapay.org/ p2p remains its own distinct provider.
Merchant names were normalized in summary tables only: bc game / Bc game → bc.game; 1win (push) → 1win; Brazino777 / Brazino 777 → Brazino 777; Melbet → melbet. Original names are preserved in detailed tables.
Field names were normalized for consistency: spelling variants (Adress → Address, document tyoe → Document type), case variants (email → Email, phone → Phone number), and Spanish variants (Numero de celular → Phone number, Mail → Email) were unified.
The cell value "CC" in the Nequi / 4bill / 1win merchant side was treated as Document number, as CC (Cédula de Ciudadanía) is a document identifier, not a standalone field name.
Merged cells in the spreadsheet were resolved: where a provider cell spanned multiple rows, the provider value was applied to all covered rows. The same was done for provider-side cells that were merged across multiple merchant rows.
This analysis is based solely on observed fields in the spreadsheet. No external knowledge, compliance assumptions, or inferred requirements were added. Some payment methods (Bre-b, Tpaga QR, Movii PSE) have very limited data and their findings should be treated as preliminary.
